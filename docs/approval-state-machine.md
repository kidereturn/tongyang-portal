# 결재 시스템 상태머신 (Approval State Machine)

버전: v2 · 작성일: 2026-04-22

## 1. 개요

이 문서는 동양 포털 내부회계 증빙 결재 시스템의 **모든 상태 전이와 경우의 수** 를 정의한다.  
3 가지 역할(담당자 / 승인자 / 관리자)의 모든 액션과 그에 따른 **증빙제출현황(홈 KPI 타일) · 증빙목록(테이블) · 상단요약(sum-strip)** 3 지점의 카운트가 동일하게 반영되어야 한다.

## 2. 핵심 상태 필드

| 필드 | 테이블 | 값 | 의미 |
|---|---|---|---|
| `submission_status` | `activities` | `미완료` / `완료` / `승인` / `반려` | 결재 **진행 단계** |
| `review_status` | `activities` | `미검토` / `검토중` / `완료` / **`수정제출`** *(신규)* | 관리자의 **별도 검토 상태** |
| `approval_requests.status` | `approval_requests` | `submitted` / `approved` / `rejected` / **`cancelled`** *(신규)* | 개별 상신 건의 결재 결과 |

## 4. 역할별 가능 액션

| 액션 | 담당자 | 승인자 | 관리자 | 영향받는 필드 |
|---|:-:|:-:|:-:|---|
| 증빙 업로드 (중간저장 포함) | ✓ | — | — | `evidence_uploads` 만 변경, 상태 유지 (`미완료`) |
| 중간저장 파일 수정/삭제 | ✓ | — | — | `evidence_uploads` 삭제/교체 |
| 결재상신 | ✓ | — | — | `submission_status`: `미완료`→`완료`, `approval_requests` row 신규 |
| 승인 | — | ✓ | — | `submission_status`: `완료`→`승인`, `approval_requests.status`=`approved` |
| 반려 | — | ✓ | — | `submission_status`: `완료`→`반려`, `approval_requests.status`=`rejected` |
| 승인자 승인 취소 | — | ✓ | — | `submission_status`: `승인`→`완료` (원래대로 환원) |
| 재상신 (반려건) | ✓ | — | — | `submission_status`: `반려`→`완료`, 새 `approval_requests` row 또는 기존 row 상태 전환 |
| **관리자 강제 취소** | — | — | ✓ | `submission_status`→`미완료`, `review_status`→`미검토`, `approval_requests.status`→`cancelled` |
| 검토상태 변경 + 저장 | — | — | ✓ | `review_status` 만 변경 (미검토/검토중/완료/**수정제출**) |
| **수정제출 저장** | — | — | ✓ | `review_status`→`수정제출`, `submission_status`→`미완료`, `approval_requests.status`→`cancelled` |

## 5. 경우의 수 전체 (state × action)

### Case A. 기본 흐름

| 시나리오 | 담당자 화면 카운트 | 승인자 화면 카운트 | 관리자 화면 카운트 |
|---|---|---|---|
| A-1. 담당자 업로드만 (중간저장) | 전체 +1, 미완료 +1 | — | 전체 +1, 미완료 +1, 미검토 +1 |
| A-2. A-1 → 담당자 상신 | 전체 +1, 상신완료 +1 | 전체 +1, 상신완료 +1 | 전체 +1, 상신완료 +1, 미검토 +1 |
| A-3. A-2 → 승인자 승인 | 전체 +1, 승인완료 +1 | 전체 +1, 승인완료 +1 | 전체 +1, 승인완료 +1, 미검토 +1 |
| A-4. A-3 → 관리자 검토완료 저장 | (변화 없음) | (변화 없음) | 완료(검토) +1, 미검토 −1 |

### Case B. 반려 흐름

| 시나리오 | 담당자 화면 | 승인자 화면 | 관리자 화면 |
|---|---|---|---|
| B-1. A-2 → 승인자 반려 | 상신완료 −1, 반려 +1 | 상신완료 −1, 반려 +1 | 상신완료 −1, 반려 +1 |
| B-2. B-1 → 담당자 재상신 | 반려 −1, 상신완료 +1 | 반려 −1, 상신완료 +1 | 반려 −1, 상신완료 +1 |
| B-3. B-2 → 승인자 승인 | 상신완료 −1, 승인완료 +1 | 상신완료 −1, 승인완료 +1 | 상신완료 −1, 승인완료 +1 |

### Case C. 관리자 강제 취소 (언제든 가능)

| 출발 상태 | 담당자 변화 | 승인자 변화 | 관리자 변화 |
|---|---|---|---|
| C-1. 상신완료에서 취소 | 상신완료 −1, 미완료 +1 | 상신완료 −1, (해당 건 안 보임) | 상신완료 −1, 미완료 +1, (검토상태 → 미검토 복귀) |
| C-2. 승인완료에서 취소 | 승인완료 −1, 미완료 +1 | 승인완료 −1, (해당 건 안 보임) | 승인완료 −1, 미완료 +1 |
| C-3. 반려에서 취소 | 반려 −1, 미완료 +1 | 반려 −1, (해당 건 안 보임) | 반려 −1, 미완료 +1 |

> 핵심: **취소 = 완전 초기화**. 담당자 입장에서 새 증빙 제출을 시작하는 것과 동일.

### Case D. 수정제출 흐름 (신규)

| 시나리오 | 담당자 변화 | 승인자 변화 | 관리자 변화 |
|---|---|---|---|
| D-1. 승인완료 상태에서 관리자 "수정제출" 저장 | 승인완료 −1, 미완료 +1, **수정제출 +1** | 승인완료 −1, (해당 건 안 보임) | 승인완료 −1, 미완료 +1, **수정제출 +1** (미검토/검토중 −1) |
| D-2. D-1 → 담당자 재상신 | 미완료 −1, 수정제출 −1, 상신완료 +1 | 상신완료 +1 | 미완료 −1, 수정제출 −1, 상신완료 +1, 검토중 +1 |
| D-3. D-1 → 관리자가 재취소 | 수정제출 −1, (여전히 미완료 상태 유지) | 변화 없음 | 수정제출 −1, (미완료는 이미 +1 된 상태로 유지), 미검토 +1 |

> 수정제출은 **담당자에게도 카운트** 가 보여야 한다 (사용자 요구사항).

### Case E. 중간저장 후 수정/삭제 (버그 수정)

| 시나리오 | 처리 |
|---|---|
| E-1. 담당자가 파일 A 중간저장 후 다른 파일 B 로 교체 | 파일 A 삭제 + 파일 B 업로드, `evidence_uploads.status='pending'` 유지 |
| E-2. 담당자가 중간저장한 파일 행 제거 | `evidence_uploads` row 삭제 + Storage 파일 삭제 |

## 6. 카운트 계산 규칙 (3 지점 동일)

**필수 원칙:** submission_status 만 보고 카운트한다. review_status 의 `수정제출` 은 별도 카운트(추가 집계).

```ts
// 모든 화면에서 동일하게 계산
const stats = {
  total:     activities.length,
  pending:   activities.filter(a => a.submission_status === '미완료').length,
  complete:  activities.filter(a => a.submission_status === '완료').length,
  approved:  activities.filter(a => a.submission_status === '승인').length,
  rejected:  activities.filter(a => a.submission_status === '반려').length,
  // 관리자 전용 (담당자는 자기 담당건만 집계)
  modifyReq: activities.filter(a => a.review_status === '수정제출').length,
}

// 합계 검증: pending + complete + approved + rejected === total
```

**수정제출 카운트 표시 대상:**
- 관리자 화면: 항상 표시
- 담당자 화면: 본인 담당 건 중 `review_status === '수정제출'` 인 것만 집계해서 표시
- 승인자 화면: 표시하지 않음 (수정제출은 승인자 이전 단계로 되돌아간 상태이므로 승인자 시야에서 사라짐)

관리자에게는 추가로 `수정제출` 뱃지 표시.

## 8. 관리자 "저장" 버튼 정책

관리자가 `review_status` select 를 바꿔도 **저장 버튼을 누르기 전에는** 실제 DB 저장이 되지 않는다.  
저장 버튼 클릭 시:

1. `review_status` 확정
2. `수정제출` 인 경우 동시에:
   - `submission_status` = `미완료`
   - 해당 건의 `approval_requests` 에서 `status`=`submitted|approved|rejected` 인 row 를 `cancelled` 로 업데이트
   - 담당자·승인자 알림 발송
3. 3 지점 카운트 동시 재계산
