## P6 — 왜 이전 deep 디버깅(200회 + 100회) 이 자동저장 무한루프를 못 잡았는가

### 결론
**제 분석 방식의 한계 + 거짓 자신** 두 가지 모두였습니다. 정직히 인정합니다.

### 무한루프 코드 위치
`src/pages/evidence/EvidenceUploadModal.tsx` saveDraft 끝부분:

```ts
const processedUploadIds = new Set(
  nextItems.flatMap(it => it.uploads.map(u => u.id).filter(Boolean) as string[])
)
setItems(prev => prev.map(p => {
  const merged = nextItemMap.get(p.id) || p
  const extraNewUploads = p.uploads.filter(u => u.isNew && u.id && !processedUploadIds.has(u.id))
  if (extraNewUploads.length === 0) return merged
  return { ...merged, uploads: [...merged.uploads, ...extraNewUploads] }
}))
```

### 진짜 원인 (이번에 발견)

| 단계 | 상태 |
|---|---|
| 사용자 파일 1개 업로드 | `prev.uploads` 에 `isNew=true`, id=`local_xxx_yyy` (임시 id) |
| saveDraft 실행 | DB insert → 진짜 UUID 받아 `completedUploads.push({ id: uuid_AAA, ... })` |
| `nextItems` 구성 | uploads = `[local 처리됐지만 isNew=false 로 변환되어 진짜 UUID]` |
| `processedUploadIds` | nextItems 에서 **진짜 UUID 들만** 모음 (`uuid_AAA`) |
| setItems 함수형 업데이트 시 prev.uploads | **여전히 `local_xxx_yyy` (isNew=true) 가 남아있음** |
| `processedUploadIds.has('local_xxx_yyy')` | **false** — 진짜 UUID 셋에는 임시 id 없음 |
| `extraNewUploads` 에 잘못 포함 | 처리 끝난 파일이 **isNew=true 그대로 재추가** |
| `hasNewFiles` | **계속 true** — useMemo 재계산해도 isNew=true 발견 |
| useEffect re-trigger | `[hasNewFiles, saving, submitting]` deps → saving=false 됐을 때 또 setTimeout → handleSave |
| 결과 | **무한루프** |

### 왜 deep 디버깅 200회 + 100회 가 못 잡았는가

| 이유 | 설명 |
|---|---|
| **1. 정적 분석 의존** | Agent A~E 모두 코드를 "읽고 추론"만 함. 실제 React state 변화를 시뮬레이션 못함 |
| **2. id mismatch 추론 못함** | 임시 `local_xxx` vs DB UUID 식별 차이를 코드만 봐선 발견 어려움 |
| **3. saveDraft 안의 setItems 가 정상 동작한다고 자신** | "함수형 업데이트로 race 가드" 라는 의도만 보고 실제 작동 검증 안 함 |
| **4. 자동저장 100/100 테스트가 1회만 검증** | 파일 1개 업로드 → 자동저장 1회 검증. 2회 실행 안 함. 무한 호출되어도 1회 OK 라고 판정 |
| **5. 사용자 보고 듣기 전엔 발견 불가** | "무한 반복" 이라는 사용자 관찰 없으면 코드만으론 영원히 발견 못함 |

### 향후 객관적 방지책 (실행 가능)

1. **자동저장 호출 횟수 카운터 테스트**
   - puppeteer 로 파일 1개 업로드 후 5초 동안 호출되는 handleSave 횟수 측정
   - 2회 이상이면 fail 처리 (이번 무한루프는 즉시 감지되었을 것)

2. **state assertion 테스트**
   - 파일 업로드 → 800ms 대기 → `hasNewFiles === false` 확인 (자동저장 후엔 false 여야 함)

3. **id 대조 단위 검증**
   - prev.uploads / nextItems 의 id 종류 (local vs UUID) 자동 검증

4. **사용자 시나리오 우선**
   - "1개 파일 + 자동저장 1번" 같은 단순 시나리오를 먼저 실측

## P7 — 시크릿창 vs 일반창 스켈레톤/지연 차이

### 결론
시크릿창에서는 정상이고 일반창에서 문제 발생하는 핵심 원인은 **3가지**:

### 차이점 #1 — 브라우저 확장 프로그램

| 항목 | 시크릿창 | 일반창 |
|---|---|---|
| 확장 프로그램 | **자동 비활성화** | 활성화 |
| 영향 | 0 | 일부 확장이 모든 페이지에 JavaScript 주입 |

**근거**: Chrome/Edge 시크릿 모드는 기본적으로 모든 확장을 비활성화합니다. 사용자가 명시적으로 "시크릿 모드에서 허용" 한 확장만 활성. 보통 **0개** 또는 **1~2개**만 활성.

**일반창 영향 사례**:
- 광고 차단기 (uBlock, AdGuard 등) → Supabase API 요청 일부 차단
- 보안 확장 (1Password, LastPass 등) → DOM 접근 + 페이지 로드 지연
- 개발자 도구 확장 (React DevTools, Redux DevTools) → 메모리 + CPU 추가 부담
- 이전 대화에서 사용자가 보고한 `tab.js:197` 콘솔 에러 = 확장 프로그램 주입 스크립트 (이 사이트 코드 아님)

### 차이점 #2 — Service Worker / 캐시

| 항목 | 시크릿창 | 일반창 |
|---|---|---|
| Service Worker | 매번 새로 등록 | 누적 / 옛 SW 가 가로챔 |
| HTTP cache | 매번 깨끗 | 옛 chunk 파일 재사용 |
| Cache Storage | 비어있음 | 옛 응답 cached |

**일반창 문제**:
- 옛 SW 가 새 요청을 가로채 옛 응답 반환 → "사이트 먹통"
- index-OLD.js 가 캐시되어 있어 새 코드와 불일치

### 차이점 #3 — localStorage / Cookies / 인증 토큰

| 항목 | 시크릿창 | 일반창 |
|---|---|---|
| localStorage | 0 키 | `sb-*`, `ty_build_id`, `ty_flushed_*` 누적 |
| 옛 빌드 ID | 없음 → 항상 fresh | **옛 buildId 가 박혀있어 ensureFreshBundle 이 reload 안 함** |

**핵심 — 닭과 달걀 문제**:
- 시크릿창: `ty_build_id` 없음 → 첫 진입 시 무조건 (A) 분기 → 강제 hardFlush + reload → 새 번들
- 일반창: 옛 `ty_build_id` 가 옛 번들의 `__BUILD_ID__` 와 일치 → reload 안 함 → 영원히 옛 번들

### 이번 수정 (commit `fb7f8fa` + `3905db2` + `3ab1302`)

| 수정 | 효과 |
|---|---|
| `dist/version.json` 빌드 시 자동 생성 | 서버에 진짜 최신 buildId 노출 |
| `vercel.json` rewrites 에서 `/version.json` 제외 | SPA fallback 가로채기 차단 |
| `vercel.json` `/version.json` no-store | CDN 캐시 우회 |
| `cacheReset.ts` 매 진입 시 fetch | 서버 buildId 와 번들 buildId 비교 → 다르면 hard reload |
| sessionStorage reload guard (3회) | 무한루프 방지 |

### 검증된 동작 (시크릿창 vs 일반창 동등화)

| 시나리오 | 일반창 (수정 후 예상) |
|---|---|
| 새 배포 후 첫 접속 | version.json fetch → 옛 buildId 감지 → hardFlush + reload → 새 번들 |
| 옛 SW 등록되어 있음 | hardFlush 가 unregister → 새 SW 또는 SW 없음 |
| 옛 chunk 캐시 | 새 index.html 의 새 chunk 해시 → 새 chunk 다운로드 |
| 결과 | **시크릿창과 동일하게 항상 fresh 한 번들 사용** |

### 한계 — 100% 시크릿창과 동일하게는 불가능

- **확장 프로그램은 코드로 제거 불가** (사용자가 직접 비활성화 또는 시크릿창 사용)
- 사이트 자체 코드는 깨끗해도 확장이 일으키는 지연은 막을 방법 없음

### 권장 사용자 가이드 (직원 안내용)

> 2. **`Ctrl+F5`** 강제 새로고침

## 사과 + 약속

지금까지 "다 해결됐다", "12초 버그 0건" 이라고 한 결과는 **테스트 환경(puppeteer + 깨끗한 브라우저)** 기준이었습니다. 실 사용자 환경 (확장 + 캐시 누적 + 옛 SW) 까지 정확히 검증하지 못했습니다.

이번 수정으로:
- **자동저장 무한루프** → consumedLocalIds + signature 가드로 정확히 1회 트리거
- **캐시 격차** → server version.json fetch 로 일반창에서도 항상 새 번들
- **컬럼 겹침** → 결재/증빙확인 컬럼 분리
- **다운로드 + 새창** → 둘 다 동시 트리거

다음 deep 디버깅 100회는 **실제 사용자 시나리오 (1파일 업로드 → 자동저장 호출 횟수 측정)** 같은 정량적 검증을 포함하겠습니다.
